# TasksBoard

個人利用を想定したタスク管理アプリです。
カテゴリや優先度でタスクを整理し、ダッシュボードで進捗を確認することができます。

## URL

https://task-app-ts.vercel.app/

ログインパスワード：`task1234`

## 機能

- ダッシュボード（完了率・カテゴリ別タスク数・優先度別未完了数・未完了タスク一覧）
- タスクの作成・編集・削除
- タスク一覧（カテゴリ・優先度・ステータスによる絞り込み・一括リセット）
- タスク一覧のソート（カテゴリ・優先度・ステータス・期限）
- タスク一覧のページネーション
- タスクへの期限設定と期限バッジ表示（期限切れ・当日・3日以内）
- カテゴリ名の編集（設定画面）
- 操作後のトースト通知（作成・保存・削除）

## 採用技術と選定理由

| 技術 | 選定理由 |
|------|---------|
| Next.js (App Router) | Reactベースのフレームワークとして標準的な構成であり、個人学習での経験もあったため採用しました。フォルダ構造がそのままURLになるApp Routerは直感的で保守しやすい点を評価しています。 |
| TypeScript | 型定義によってデータ構造が明確になり、変更やリファクタリングを安全に行えるため採用しました。型の一元管理により、修正が必要な箇所をすぐに特定できます。 |
| Tailwind CSS | 実務や個人学習での経験があったため採用しました。クラス名だけでスタイルを書けるためCSSファイルを別途管理する手間が省けますし、レスポンシブ対応もシンプルに書けるところを評価しています。 |
| localStorage | サーバーを用意せずにデータを保存できるlocalStorageが今回の要件に最もシンプルだと判断して採用しました。データ処理をカスタムフックに集約しており、将来データベースに移行する場合もUI側のコードを変えずに対応できる設計にしています。 |
| Vercel | 個人利用での経験があったため採用しました。GitHubへのプッシュだけで自動デプロイされるためCI/CDの手間が省けて楽な点を評価しています。 |

## 工夫した点

**useReducerによるタスク・カテゴリ状態管理**

タスクとカテゴリの状態管理に`useReducer`を採用しました。
`useState`で個別の関数（追加・更新・削除など）を管理する場合、操作が増えるたびに関数も増えていきます。`useReducer`を使うことで「どんな操作が起きたか」をアクションとして明示し、全ての状態変更を`reducer`という1つの関数に集約しています。操作の種類が増えても`reducer`に1ケース追加するだけで済む設計です。

**React.memoによるTaskCardの再レンダリング最適化**

タスク一覧では`TaskCard`コンポーネントを`React.memo`でラップしています。
Reactはデフォルトで親コンポーネントが再レンダリングされると子コンポーネントも全て再レンダリングします。たとえば1件のタスクを完了にしたとき、関係のない他のタスクカードまで再レンダリングが走ります。
`React.memo`を使うと渡されたpropsが前回から変わっていない場合はレンダリングをスキップするため、変化のあったカードだけが更新されます。

**useMemoによるフィルタ・ソート処理の最適化**

タスク一覧のテキスト検索・カテゴリ・優先度・ステータスの絞り込みとソート処理を`useMemo`でまとめています。
`useMemo`を使わない場合、画面のどこかに変化があるたびに全件に対してフィルタ・ソート処理が毎回走ります。`useMemo`を使うことで、フィルタ条件やタスクデータに変化があったときだけ再計算されるようになります。
また絞り込み条件が変わった際はページが自動的に1に戻るよう制御しており、意図しない空ページが表示されないよう考慮しています。

**Context APIによるカテゴリ状態管理**

カテゴリデータはContext APIで管理し、どの画面からでも同一のデータにアクセスできる設計にしました。
Context APIを使わない場合、親から子へpropsでデータを渡し続ける必要があり、階層が深くなるほどコードが煩雑になります（propsのバケツリレーと呼ばれます）。Contextに置くことで必要な画面が直接データを取得できます。
カスタムフックとContextをセットで管理しており、将来的にタスクデータも同じ構成に移行しやすい設計にしています。

**型定義・定数の一元管理**

`types/index.ts` に型定義と表示用の定数（優先度ラベル・カテゴリカラーなど）をまとめています。
優先度やカテゴリの色・表示名を直接文字列で書かず定数を参照する形にすることで、変更が必要な場合も1ファイルの修正で全体に反映されます。

**フォームのバリデーションと差分チェック**

タスクの作成・編集画面では、タスク名の空文字チェックをフォーム送信時に実行し、エラーメッセージをフィールド直下に表示します。タスク名は最大50文字とし、入力欄に文字数カウンターを表示しています。
編集画面では元のデータとの差分を検出し、変更がない場合は保存ボタンを非活性にすることで不要な操作を防いでいます。

**操作後のトースト通知**

タスクの作成・保存・削除後にトースト通知を表示し、操作が完了したことをフィードバックします。URLのクエリパラメータでメッセージを渡し、遷移先のタスク一覧画面で受け取って表示する設計にしています。

**削除操作の確認ダイアログ**

タスクの削除は取り消しができない操作のため、削除ボタンを押しただけでは削除されず、確認ダイアログを挟む設計にしました。誤操作によるデータ消失を防ぐための配慮です。

**レスポンシブ対応**

PC・スマートフォン両方での操作を考慮し、PCではサイドバー、スマートフォンではボトムナビゲーションを表示する構成にしました。

## セットアップ手順

```bash
npm install
npm run dev
```

ブラウザで http://localhost:3000 を開いてください。

## 既知の問題

スマートフォンのブラウザ（iOS Safari）でデプロイ環境にアクセスした際、ブラウザのツールバーによってボトムナビが隠れてしまう場合があります。ローカル環境では正常に動作することを確認しています。
